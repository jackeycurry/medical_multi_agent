## 一、项目架构

### 1.1 总览

```
┌──────────────────────────┐       ┌──────────────────────────────┐
│   前端 (React 19 SPA)     │       │   后端 (FastAPI + SQLite)      │
│   localhost:5173         │◄─────►│   localhost:8000              │
│                          │  /api │                               │
│   Vite dev proxy ────────┘       │   ┌─ routers/   (路由层)       │
│                                  │   ├─ services/  (业务逻辑)     │
│                                  │   ├─ models/    (ORM模型)      │
│                                  │   ├─ schemas/   (Pydantic)     │
│                                  │   └─ middleware/ (JWT/API Key) │
│                                  │                               │
│                                  │   DashScope API (通义千问)      │
│                                  │   qwen-plus / OpenAI兼容模式    │
└──────────────────────────┘       └──────────────────────────────┘
```

### 1.2 后端分层架构 (backend/)

```
请求 → routers/  → services/  → models/
  │      薄层路由    业务逻辑     SQLAlchemy ORM
  │      参数校验    编排调用     SQLite 持久化
  │
  ├─ middleware/auth.py     JWT 认证守卫
  ├─ middleware/api_key.py  X-API-Key 鉴权 + 限流
  └─ schemas/               Pydantic 请求/响应模型
```

**核心设计原则**：
- 路由层不写业务逻辑，只做参数校验 + 调用 service + 返回响应
- Service 层接收 SQLAlchemy Session，自行管理事务
- 流式 SSE 落库使用独立 `SessionLocal()`，不复用请求级 db

### 1.3 前端分层架构 (frontend/src/)

```
App.tsx (React Router v7)
  ├─ AuthContext      → JWT 状态、login/register/logout
  ├─ api/client.ts    → Axios 实例 (自动附加 Bearer token, 401 清除)
  ├─ layouts/         → 角色布局壳 (PatientLayout / DoctorLayout / HospitalLayout)
  ├─ pages/           → 按角色分目录
  │   ├─ admin/       → super_admin: Dashboard, 医生CRUD, 调用日志
  │   ├─ patient/     → 医院选择 → 症状输入 → AI对话(SSE) → 转接真人
  │   ├─ doctor/      → 排队列表 → 接单 → 对话(长轮询) + AI建议
  │   └─ hospital/    → 统计、患者列表、会话管理
  └─ components/      → ChatBubble, ChatInput, DoctorCard, StatCard
```

## 三、认证与鉴权

### 3.1 JWT 认证 (内部 API)

```
Authorization: Bearer <JWT>
```

JWT Payload: `{ sub: user_id, role: "patient"|"doctor"|..., hid: hospital_id, exp: ... }`

中间件链：
- `get_current_user` — 解码 JWT，无效抛 401
- `get_optional_user` — 同上但无效时返回 None（不抛异常）
- `require_role(*roles)` — 工厂函数，校验 role 是否在白名单
- `require_hospital` — 返回 hospital_id，super_admin 返回 0（不过滤）

### 3.2 API Key 鉴权 (对外 API v1)

```
X-API-Key: <key>
```

配置格式（`.env`）：`key:hospital_id` 逗号分隔，如 `demo_abc:1,demo_def:2`
- 每个 key 绑定一个医院，强制多租户隔离
- 内存滑动窗口限流（每 key 每分钟封顶，默认 60 次，超出返回 429 + Retry-After）

### 3.3 路由权限矩阵

| 路由前缀 | 鉴权方式 | 角色 |
|---|---|---|
| `/api/auth/*` | 无需 token（login/register） | 公开 |
| `/api/patient/*` | JWT | patient |
| `/api/doctor/*` | JWT | doctor |
| `/api/hospital/*` | JWT | hospital_admin（super_admin 也放行） |
| `/api/doctors/*`、`/api/consultations/*` 等 | JWT | super_admin |
| `/api/v1/*` | X-API-Key | 外部调用方（按 key 绑定的医院隔离） |

## 四、核心业务流程

### 4.1 AI 问诊流程（患者端）

```
1. 患者选择医院 → GET /api/patient/hospitals
2. 输入症状 → POST /api/patient/consultations/start
   └─ 后端：routing_service.match_doctor（症状关键词 → 科室 → AI医生）
   └─ 后端：调用 ai_service.chat 生成 greeting（非流式）
3. 患者发消息 → POST /api/patient/consultations/{id}/message
   └─ 返回 SSE (text/event-stream)
   └─ 前端：fetch + ReadableStream 分帧解析 delta，逐字追加到气泡
   └─ 特殊情况：若真人医生已接管 (escalation in_progress)，返回 JSON 提示
```

**SSE 帧格式**：
```
data: {"delta": "您"}
data: {"delta": "好"}
data: {"done": true}
```

**SSE 实现细节**：
- 前端用 `fetch` + `ReadableStream`（不能用 axios/EventSource——axios 不支持流，EventSource 不支持 POST+Authorization）
- 按 `\n\n` 分帧，`buffer.split` + `frames.pop()` 保留残帧
- 生成器完成后用独立 `SessionLocal()` 写库

### 4.2 真人医生接管流程

```
患者点击"请求真人医生"
  → POST /api/patient/consultations/{id}/escalate
  → 创建 EscalationQueue (status=pending)
  → 前端每 3s 轮询 escalation-status

医生端查看排队列表
  → GET /api/doctor/queue
  → 点击"接单" → POST /api/doctor/queue/{id}/claim
  → CAS 更新（WHERE status=pending），防并发抢单

医生接入后：
  → 患者发消息 → 不调 AI，返回"真人医生正在处理中"
  → 医生回复 → POST /api/doctor/sessions/{id}/reply (source=human)
  → 患者端长轮询 /messages?after_id=X&wait=25 拉取医生回复
  → 医生可结束接诊 → POST /api/doctor/sessions/{id}/resolve
```

**真人回复后的 AI 行为**：
- 真人结束后，患者再发消息恢复调 AI
- system_prompt 被追加"不可与[真人医生]内容矛盾"约束
- 真人消息以 `[真人医生] xxx` 前缀拼入历史

### 4.3 AI 建议回复（医生端）

```
医生进入会话页面
  → GET /api/doctor/sessions/{id} → 返回 detail + ai_suggestion
  → 后续：长轮询 /wait 拿新消息，拿到后异步调 /suggestion 刷新建议

AI 建议缓存策略：
  - 缓存键：(session_id, last_message_id)
  - 消息无变化时直接命中缓存，不调 DashScope
  - 医生回复 / 新患者消息 → last_message_id 变化 → 触发重算
  - 进程内存缓存，重启丢失（可接受）
```

### 4.4 长轮询机制（替代旧 3s 间隔轮询）

**患者端** — `GET /patient/consultations/{id}/messages?after_id=X&wait=25`
- 服务端循环 0.2s sleep 查询，有 `id > after_id` 的新消息立即返回
- 超时（25s 后）返回空数组，前端立即重连
- 前端按 `message.id` Set 去重，防止 handleSend 和长轮询重复

**医生端** — `GET /doctor/sessions/{id}/wait?after_id=X&wait=25`
- 同上机制，只返 messages 不含 suggestion（suggestion 异步单独拉）
- 前端按 id 去重，仅真正 append 时才触发 `refreshSuggestion()`

### 4.5 科室路由匹配

```
routing_service.match_doctor(symptom, department, hospital_id)
  1. 如果指定 department → 精确匹配
  2. 症状关键词 → 映射到科室
     心内科: [胸闷,胸痛,心慌,心悸,...]
     内分泌科: [血糖,糖尿病,甲亢,...]
     儿科: [发烧,发热,咳嗽,小孩,...]
     ...
  3. fallback: 全科 → 任意在线医生
```

### 6.5 生产部署注意事项

- **CORS**：`main.py` 当前 `allow_origins=["*"]` + `allow_credentials=True` 在生产中不合法，应改为具体域名
- **限流**：内存计数器，多 worker 需迁移 Redis
- **数据库**：SQLite 单文件，建议迁 PostgreSQL + Alembic 迁移
- **前端构建**：`npm run build` → `frontend/dist/` → 用 nginx 反代 `/api` 到后端
- **JWT_SECRET**：必须换强随机值，不要用 `.env.example` 里的示例

### 6.6 数据库重置

```bash
# 关闭后端 → 删除 db 文件 → 重启（自动 seed）
rm backend/data/ai_doctor.db
python run.py
```

## 七、关键技术决策记录

| 决策 | 原因 |
|---|---|
| AI 建议缓存（进程内存） | 避免医生页面每 3 秒轮询重复消耗 DashScope 配额 |
| 长轮询替代短间隔轮询 | 患者/医生消息延迟从 ~1.5s 降到 ~250ms |
| 医生 reply 异步化 | reply 写库即返回，AI 建议由前端异步拉 `/suggestion`，UI 不阻塞 |
| 抢单用 SQL CAS | `UPDATE ... WHERE status='pending'`，rowcount=0 即被抢，无需应用层锁 |
| SSE 写库用独立 SessionLocal | 避免生成器内在请求 db 已关闭后写库失败 |
| 外部 API key:hid 绑定 | 强制多租户隔离，每个 key 只能访问所属医院的医生和会话 |
| fetch+ReadableStream 解析 SSE | EventSource 不支持 POST+Authorization，axios 不支持流 |
| 按 id Set 去重 | handleSend 和长轮询可能同时拿到同一批消息，靠 message.id 去重 |
